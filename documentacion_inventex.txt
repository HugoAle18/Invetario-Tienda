### 1.2 Conexión entre Capas

**Frontend (React + Vite)** -> **Backend (Node.js + Express)** -> **Base de Datos (Supabase PostgreSQL)**

1. Frontend: React con Vite compila JSX a HTML/JS estático. Se comunica exclusivamente con el backend via HTTP (Axios). No hay cliente Supabase en frontend por seguridad.

2. Backend: Express.js recibe peticiones REST en /api/v1. Cada ruta pasa por:
   - Middleware authenticate (verifica JWT)
   - Middleware authorize (verifica rol cuando es necesario)
   - Validacion con express-validator
   - Controlador que orquesta la logica
   - Modelo que ejecuta la query contra Supabase PostgreSQL

3. Base de Datos: Supabase PostgreSQL alojada en la nube. El backend se conecta con @supabase/supabase-js usando la SUPABASE_SERVICE_ROLE_KEY (rol service_role, omite RLS). Las operaciones complejas (entrada/salida con actualizacion de stock) se ejecutan como funciones RPC de PostgreSQL para garantizar atomicidad.

### 1.5 Flujo de Autenticacion

Paso 1 - Login: El usuario ingresa email y contrasena en LoginPage.jsx. El formulario esta validado con Zod (email valido, password no vacio).

Paso 2 - Peticion POST: AuthContext.login() llama a authApi.login({ email, password }), que envia POST a /api/v1/auth/login.

Paso 3 - Validacion backend: authValidator.js valida formato con express-validator. Si falla, retorna 400.

Paso 4 - Consulta a BD: authController.login() busca el usuario por email en usuarios:
SELECT id, nombre, email, password, rol, activo FROM usuarios WHERE email = ?
Si no encuentra el usuario o esta inactivo, retorna 401 "Credenciales invalidas".

Paso 5 - Verificacion de contrasena: Compara el password ingresado con el hash almacenado usando bcrypt.compare(). Si no coincide, retorna 401.

Paso 6 - Generacion de tokens: Crea un JWT (accessToken) con jsonwebtoken.sign() que contiene { id, email, rol, nombre } y expira en 1 hora. Crea un refreshToken aleatorio (40 bytes hex), lo guarda en refresh_tokens con expiracion de 7 dias.

Paso 7 - Respuesta: Backend devuelve { accessToken, refreshToken, user }.

Paso 8 - Almacenamiento: AuthContext.login() guarda el accessToken en memoria (variable accessToken en axios.js) y el refreshToken en localStorage mediante token.js.

Paso 9 - Redireccion: LoginPage usa navigate() a /admin/dashboard o /empleado/panel segun user.rol.

Paso 10 - Peticiones autenticadas: El interceptor de Axios en axios.js agrega Authorization: Bearer <accessToken> a cada request. Si recibe 401, automaticamente intenta refrescar el token llamando a /auth/refresh, y si falla, redirige a /login.

| Pantalla | Funcionalidades |
|----------|----------------|
| Panel | KPIs: total productos, movimientos del mes, alertas de stock. Tabla de movimientos recientes del empleado. |
| Stock | Consulta de productos con stock actual (solo lectura). Badge rojo para stock bajo. Tabla con headers gradient. |

| Salida | Registrar salida de producto: buscador con autocomplete, cantidad, motivo. Valida stock suficiente. Notificacion automatica. |
| Movimientos | Historial de movimientos del empleado. Mismos filtros que admin pero solo sus propios movimientos. |

**Restricciones del empleado vs admin:**
- No ve ni edita usuarios
- No ve precios en movimientos (backend filtra campo precio_unitario)
- No puede crear/editar/eliminar productos, categorias, proveedores
- Solo ve sus propios movimientos (backend filtra por usuario_id)
- No tiene acceso a reportes analiticos
- No tiene acceso a configuracion del sistema

### Funcionalidades Generales

**Modo Oscuro/Claro:**

## SECCION 4 - ERRORES Y CORRECCIONES

### 4.1 Errores de Base de Datos

ERROR: Consultas directas a Supabase fallaban con timeout de rate limit
CAUSA: Las RPC de Supabase se llamaban con sintaxis incorrecta. El modelo original usaba supabase.rpc('registrar_movimiento', ...) pero la funcion no existia con ese nombre exacto.
SOLUCION: Se reemplazaron las llamadas RPC por queries SQL directas con supabase.from('movimientos').insert(...) combinadas con supabase.rpc para registrar_entrada y registrar_salida.
LECCION: Siempre verificar que los nombres de funciones RPC en PostgreSQL coincidan exactamente con los usados en el codigo.

ERROR: Seed de contrasenas incorrecto
CAUSA: El script seed.js insertaba contrasenas en texto plano en lugar de hash bcrypt, y las contrasenas de los usuarios por defecto no funcionaban al hacer login.
SOLUCION: Se actualizo seed.js para generar los hashes con bcrypt antes de insertar. Se agrego un UPDATE correctivo para los usuarios ya existentes.
LECCION: Los datos demo deben generarse con las mismas transformaciones que los datos reales (hashing de contrasenas).

### 4.2 Errores de Autenticacion

ERROR: CORS bloqueaba peticiones del frontend
CAUSA: El backend no tenia configurado correctamente CORS para el origen de Vercel ni para localhost:5173. Helmet tambien bloqueaba la politica de recursos cruzados.
SOLUCION: Se configuro CORS con lista de origenes permitidos y credentials: true. Se deshabilito crossOriginResourcePolicy en Helmet.
LECCION: En arquitecturas desacopladas, CORS debe configurarse temprano para evitar bloqueos de depuracion.

ERROR: Interceptor de Axios y refresh token en rutas de auth
CAUSA: El interceptor de Axios intentaba refrescar token incluso en las rutas /auth/refresh y /auth/login, causando bucles infinitos.
SOLUCION: Se agrego la funcion isAuthRoute() que excluye las rutas que contienen /auth/ del interceptor 401.
LECCION: Los interceptores automaticos deben tener guards para evitar recursion en endpoints de autenticacion.

ERROR: Rate limiter bloqueaba logins legitimos
CAUSA: El rate limiter estaba configurado con un maximo muy bajo (10-50 peticiones) y usaba req.ip como key, pero tras un proxy (Render) todas las IPs aparecian como la misma.
SOLUCION: Se incremento el limite a 200, se cambio la key a req.body?.email, y se agrego app.set('trust proxy', 2). Finalmente se deshabilito el rate limiter temporalmente.
LECCION: Los rate limiters detras de proxies requieren trust proxy y una estrategia de keying adecuada.

### 4.3 Errores de Despliegue

ERROR: React Router fallaba en Vercel con 404 en rutas anidadas
CAUSA: Vercel no reconoce las rutas SPA de React. Al recargar /admin/productos, devolvia 404.
SOLUCION: Se agrego vercel.json con rewrite que mapea todas las rutas a /index.html:
{ "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }] }
LECCION: Toda SPA desplegada en Vercel necesita rewrites para funcionar correctamente.

ERROR: Backend de Express fallaba en Vercel por falta de configuracion serverless
CAUSA: Vercel espera una funcion serverless, pero el backend usaba un servidor HTTP tradicional.
SOLUCION: Se agrego vercel.json en backend con @vercel/node como builder para convertir Express en serverless.
LECCION: Express necesita configuracion especifica para desplegarse como serverless en Vercel.

ERROR: Chunks de JS demasiado grandes (>500KB)
CAUSA: Recharts y dependencias de graficos aumentaban el bundle size.
SOLUCION: Se considero dynamic import, pero no se implemento (solo advertencia, no error).
LECCION: Monitorear el tamano del bundle en produccion para evitar tiempos de carga largos.

### 4.4 Errores de la IA (Alucinaciones)

ERROR: OpenCode mezclo sintaxis de MySQL con PostgreSQL
CAUSA: Al generar queries SQL para Supabase, la IA uso funciones de MySQL (como NOW() sin DEFAULT correcto, o tipos como VARCHAR en lugar de TEXT).
SOLUCION: Se revisaron y corrigieron todas las queries para usar sintaxis PostgreSQL estandar.
LECCION: La IA tiende a mezclar dialectos SQL. Siempre revisar las queries generadas contra la documentacion de PostgreSQL.

ERROR: OpenCode genero imports de librerias inexistentes
CAUSA: En algunos componentes, la IA importaba funciones de modulos que no existian o usaba nombres incorrectos en lucide-react.
SOLUCION: Se corrigieron manualmente los imports y se verifico cada build.
LECCION: Los imports generados por IA deben verificarse contra el package.json real antes de ejecutar.

ERROR: La IA genero codigo de autenticacion con Supabase Auth en frontend
CAUSA: OpenCode asumio que el frontend tendria acceso directo a Supabase (usando @supabase/supabase-js en el cliente), cuando la arquitectura definida era que todo pasara por el backend Express.
SOLUCION: Se elimino el cliente Supabase del frontend y se reemplazo por llamadas API a traves de Axios.
LECCION: La IA a veces ignora restricciones arquitectonicas explicitas. Los prompts deben repetir las reglas clave.

ERROR: La IA genero el rate limiter con validacion (validate: X-Forwarded-For)
CAUSA: El rate limiter de Express usaba validate: { XForwardedForHeader: false } pero la opcion correcta habia cambiado en la version instalada.
SOLUCION: Se elimino la opcion validate y se configuro correctamente trust proxy.
LECCION: La documentacion de librerias cambia entre versiones. La IA puede generar codigo para versiones antiguas.

### 4.5 Errores de Diseno/UI

ERROR: Modo claro dejaba texto invisible en paginas de empleado
CAUSA: Los componentes de empleado solo tenian clases text-white sin equivalentes en modo claro (text-gray-900, dark:text-white).
SOLUCION: Se agregaron clases text-gray-900 (o equivalentes) con el prefijo dark: para que el modo claro tuviera texto legible.
LECCION: Cada vez que se usa un color de texto, debe tener su par claro/oscuro explicito.

ERROR: Modales y formularios no respondian al cambio de tema
CAUSA: Modal.jsx y los formularios (ProductoForm, UsuarioForm, etc.) usaban clases fijas sin variante dark:.
SOLUCION: Se cambiaron todos los modales a bg-gradient-to-b from-white to-slate-50 dark:from-slate-900 dark:to-slate-950. Los inputs se cambiaron a bg-white dark:bg-slate-800 text-gray-900 dark:text-white.
LECCION: Todos los componentes deben tener variante dark: desde el inicio. Agregarlo despues es mas costoso.

ERROR: Scroll de contenido cortaba la paginacion
CAUSA: El layout usaba h-screen en el contenedor derecho, lo que limitaba la altura y cortaba la paginacion al fondo.
SOLUCION: Se cambio a min-h-screen, se agrego pb-20 en <main> y mb-10 en los wrappers de tabla.
LECCION: h-screen limita el scroll vertical. Para layouts con header fijo y scroll en contenido, usar min-h-screen.

ERROR: Efecto glass no se notaba contra fondos blancos
CAUSA: El fondo del layout usaba colores solidos (bg-slate-50, bg-slate-950) que opacaban el efecto de backdrop-blur en los componentes glass.
SOLUCION: Se cambio el fondo del layout a un gradiente (bg-gradient-to-br from-slate-50 via-slate-100 to-blue-100/50) y los componentes glass a bg-white/40 o bg-slate-950/40 con backdrop-blur-xl para que el gradiente se viera a traves de ellos.
LECCION: El efecto glassmorphism requiere un fondo texturizado (gradiente, imagen, patron) para que el backdrop-blur sea visible.

### 4.6 Errores de Logica de Negocio

ERROR: Al editar un producto, la fila se desvanecio (flicker)
CAUSA: handleSubmit llamaba a fetchProductos() despues de actualizar, lo que recargaba toda la lista desde el servidor y causaba un re-render completo con perdida de posicion visual.
SOLUCION: Se cambio a setProductos(prev => prev.map(p => p.id === editing.id ? { ...p, ...updated } : p)) que actualiza solo el elemento editado manteniendo su posicion exacta en el array.
LECCION: Para mutaciones individuales en listas, el estado local debe actualizarse con .map() o findIndex() en lugar de recargar todo el listado.

ERROR: Productos sin stock se mostraban como seleccionables en salidas
CAUSA: ProductoSearch.jsx permitia seleccionar productos con stock 0 en modo salida.
SOLUCION: Se agrego la validacion if (tipo === 'salida' && item.stock_actual <= 0) return en la funcion selectItem() y se agrego clase opacity-40 cursor-not-allowed para productos sin stock.
LECCION: Las validaciones de reglas de negocio deben implementarse tanto en frontend como en backend (la RPC registrar_salida ya valida stock).

ERROR: El conteo de total de productos no se actualizaba al crear
CAUSA: El total (usado en el badge y paginacion) solo se actualizaba con fetchProductos(). Al crear un producto sin recargar, el total quedaba desactualizado.
SOLUCION: Se agrego setTotal(prev => prev + 1) despues de crear un producto.
LECCION: Los contadores derivados deben actualizarse sincronizadamente con las mutaciones del listado.

ERROR: Empleado veia precios de productos en movimientos
CAUSA: El backend no filtraba el campo precio_unitario para empleados en el listado de movimientos.
SOLUCION: Se agrego en movimientoController.listar(): si el usuario es empleado, se mapean los resultados eliminando precio_unitario.
LECCION: La segregacion de datos por rol debe implementarse en el backend, no confiar solo en el frontend.

### 6.2 Funciones RPC de PostgreSQL

#### registrar_entrada(p_producto_id, p_cantidad, p_motivo, p_precio_unitario, p_usuario_id)
Descripcion: Registra una entrada de producto en una transaccion atomica.
1. Valida que el producto exista y este activo (SELECT nombre FROM productos WHERE id = p_producto_id AND activo = TRUE)
2. Valida que la cantidad sea mayor a 0
3. Inserta registro en movimientos con tipo='entrada'
4. Actualiza productos.stock_actual = stock_actual + p_cantidad
5. Retorna JSONB con { exito: true, mensaje: "Entrada registrada correctamente" } o { exito: false, error: "..." }

#### registrar_salida(p_producto_id, p_cantidad, p_motivo, p_usuario_id)
Descripcion: Registra una salida de producto validando stock suficiente.
1. Obtiene nombre y stock_actual del producto
2. Valida que el producto exista y este activo
3. Valida que la cantidad sea mayor a 0
4. Valida que stock_actual >= p_cantidad; si no, retorna error con valores disponibles
5. Obtiene precio_venta del producto y lo guarda en el movimiento
6. Inserta registro en movimientos con tipo='salida'
7. Actualiza productos.stock_actual = stock_actual - p_cantidad
8. Retorna JSONB con resultado

#### obtener_alertas_stock()
Descripcion: Retorna todos los productos con stock menor o igual al minimo, ordenados por diferencia ascendente (mas criticos primero). Util para el dashboard y panel de empleado.

#### obtener_dashboard_admin()
Descripcion: Calcula y retorna en un JSONB los KPIs del dashboard: total_productos, total_categorias, total_proveedores, total_movimientos_30d, total_usuarios, valor_inventario (suma de precio_compra * stock_actual), productos_bajo_stock.

#### obtener_movimientos_recientes(p_limite DEFAULT 20)
Descripcion: Retorna los ultimos movimientos con nombres de producto y usuario, uniendo movimientos con productos y usuarios.

#### crear_notificacion_movimiento() (trigger)
Descripcion: Funcion llamada por el trigger trg_crear_notificacion_movimiento despues de cada INSERT en movimientos. Crea notificaciones para:
- Todos los administradores (excepto quien realizo el movimiento)
- El usuario que realizo el movimiento

El titulo y descripcion varian segun el tipo (entrada/salida).
